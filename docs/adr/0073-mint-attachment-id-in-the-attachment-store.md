# Mint the Attachment ID in the Attachment Store during staging

## Status

Accepted (2026-07-20).

Refines
[ADR 0059](0059-store-accepted-inbound-attachments-as-immutable-per-turn-records.md).

ADR 0059 decided that an accepted attachment occurrence has one globally unique
opaque **Attachment ID** with no Conversation, Turn, Project, Surface, filename,
path, media type, digest, provider-key or timestamp semantics, and originally
stated that the **control plane** generates it. This ADR keeps every one of
those guarantees but moves the *point of minting* into the **Attachment Store**
during staging. It changes which component owns identity generation, not what
the identity is.

## Context

An Attachment Record is published to the Attachment Store (provisional, then
accepted) before its owning Turn Receipt is authoritatively committed. The
control plane commits only a content-free batch commitment — the Store identity,
batch ID, record count and manifest digest — and never stores Records or Blobs
(ADR 0059). Minting the Attachment ID in the control plane would therefore
require the control plane to either round-trip an identity into the Store during
staging or hold per-attachment identity state it otherwise never needs.

The original ADR 0059 wording ("generated by the control plane") was, at one
point, edited in place to read "minted by the Backend / Attachment Store" so the
document matched the code. That in-place rewrite of an Accepted ADR's decision
violates the repository's history-preserving rule (`docs/adr/README.md`), which
requires a new ADR when the decision itself changes. This ADR is that record;
ADR 0059's original wording has been restored.

## Decision

The **Attachment Store mints the Attachment ID during staging**, because the
Store owns the Record and already writes the staging row. The control plane
binds only the content-free batch commitment (Store identity, batch ID, record
count, manifest digest) and learns no per-attachment identity semantics.

All ADR 0059 identity guarantees are unchanged and independent of the minting
site:

- the ID is globally unique and opaque;
- it carries no Conversation, Turn, Project, Surface, filename, path, media
  type, content digest, provider resource key or timestamp semantics;
- a duplicate Turn still observes the original ordered Records, and a later Turn
  or Run may cite an existing Attachment ID without moving, copying, relabeling
  or changing ownership.

Moving minting into Turn acceptance would widen the control plane's attachment
knowledge with no invariant gained. Should a future slice need to expose a
proposed Attachment ID *before* any byte is read, that concrete requirement —
not layering preference — is what would justify revisiting this decision in a
further ADR.

## Consequences

- The control plane's attachment surface stays minimal: one content-free batch
  commitment per Turn, no per-attachment identity state.
- Staging, acceptance and duplicate observation are unchanged; this ADR
  documents the ownership that the implementation already has.
- ADR 0059 remains the authoritative record of the immutable per-Turn Record
  model and links forward here for the identity-minting site.

## Rejected alternatives

- **Mint the Attachment ID in the control plane (the original ADR 0059
  wording).** Rejected because it forces the control plane to carry or
  round-trip per-attachment identity that it otherwise never needs, without
  strengthening opacity, uniqueness or the absence of semantics.
- **Silently rewrite ADR 0059 to say the Store mints the ID.** Rejected because
  editing an Accepted ADR's decision in place erases the historical record; the
  history-preserving rule requires this successor ADR instead.
