# Persist restart-resilient transcripts by mirroring the in-memory derived state (P-state) into a dedicated write-through `transcripts.db` and rehydrating once on cold-start miss, without ever reloading per-turn

## Status

Accepted (2026-07-06). Shipped to `main` via PR #29 (transcript store + lazy rehydrate) and PR #30 (the `clear_conversation` fan-out seam); the surface wiring that routes every `/clear` through that seam lands alongside this status update. Resolves the previously-deferred **F14** (raw transcript durability across restart) — using a write-through derived-state mirror, explicitly **not** the per-turn-reload approach that review rejected. Extends `docs/CONTEXT.md` §"Prompt Prefix & Caching" vocabulary; preserves ADR 0024's prefix-cache invariants in full.

**Implementation status (2026-07-06).** Decision 4's lazy-on-miss rehydrate is confirmed as shipped (`TranscriptStore.get_or_create` rehydrates only on an in-memory miss, never per-turn — `omicsclaw/runtime/storage/transcript.py`). Decision 6's fan-out is now routed through a single `clear_conversation(chat_id)` seam (`omicsclaw/runtime/agent/state.py`) that every surface (channels, CLI/TUI, interactive REPL) calls, so no surface can clear one store and orphan the other. The retention/GC of tool blobs + `records_by_chat` remains an **Open** pre-launch item (below).

## Context

The raw, turn-by-turn transcript lives **only** in process memory. `TranscriptStore` is a dict plus an LRU access map (`omicsclaw/runtime/storage/transcript.py:416-432`: `messages_by_chat: dict[int | str, list[dict]]` and `access_by_chat`), with no backing store. A restart or redeploy loses every conversation's raw history.

Two adjacent stores already survive a restart, which makes the transcript the lone gap:

- **Distilled memory** — the graph `memory.db` behind `omicsclaw/memory/database.py` (SQLAlchemy async engine) — persists the condensed/ranked memory that feeds the Session prefix snapshot and Volatile context, but not the raw turns.
- **Tool-result blobs** — `ToolResultStore` (`omicsclaw/runtime/storage/tool_result.py`) writes large outputs to per-chat files and references them from messages as `full_result_path` (`tool_result.py:225`), so blob *bodies* survive; but the messages that reference them do not.

For the single-user desktop path this was tolerable and deliberately deferred (F14 was gated on "multi-user channels / restart resilience becomes a goal"). That gate is now met. A restart-resilient, multi-user backend must be able to bring a conversation back after a graceful redeploy, byte-for-byte, without violating ADR 0024's **Stable prefix invariant** — which the naive fix (re-fetch history from a store every turn) would break by reserializing the **Prompt prefix** on every request.

A second constraint hardens the design: context collapse folds old messages into a frozen `system` summary, and that summary can be **LLM-generated** rather than deterministically templated. That path — the condensed collapse summary — is Decision 5 of the sibling ADR 0039 (token-native context budget), where **default-ON is the ratified target** (2026-07-03), gated on a content-fidelity-gate prerequisite before it ships ON. Whenever the LLM summary is enabled, the collapse output — and therefore the persisted summary bytes — is non-deterministic. Re-deriving it on load would produce different bytes and cost an LLM call. The store must therefore persist the *result*, not a recipe to recompute it, so that byte-identical rehydrate holds — persist-the-result is mandatory whenever the summary is enabled, not a defensive margin.

## Decision

Persist the transcript as a durable, write-through mirror of the in-memory **derived state (P-state)** — the exact bytes the store currently holds — and read it back exactly once per conversation, on an in-memory miss. Six resolutions:

1. **Persist the derived state (P-state), not a raw event log.** Mirror whatever `TranscriptStore` currently holds for a chat, including the exact, possibly LLM-generated, collapse-summary bytes emitted by `replace_history` (`transcript.py:448`, fed from `query_engine.py:637`). Do **not** persist a raw log and re-derive collapse/sanitize on load: once the opt-in LLM collapse summary (ADR 0039 Decision 5) is enabled, re-summarizing is non-deterministic and would break byte-identical rehydrate and cost an LLM call. The store's job is durability of the current state, not event sourcing.

2. **A dedicated `transcripts.db` SQLite, separate from the graph `memory.db`.** Keep the Memory-System domain (`omicsclaw/memory/database.py`) clean. The mirror uses the **standard-library `sqlite3`** driver (sync — matching `TranscriptStore`'s synchronous API; see Decision 5), so it adds **no** dependency and no async/await plumbing to the store; the graph `memory.db` keeps its own async `aiosqlite` engine, unaffected. Store one row per message with an **opaque JSON payload**, preserving the exact message shape — including tool references like `full_result_path` — without the DB understanding message semantics. Schema:
   - `transcript_chats(chat_key TEXT PK, chat_id_json TEXT, updated_at INT, last_seq INT)`
   - `transcript_messages(chat_key TEXT, seq INT, payload_json TEXT, PRIMARY KEY(chat_key, seq), FOREIGN KEY(chat_key) → transcript_chats ON DELETE CASCADE)`

   `chat_key` is **type-tagged** because chat ids are `int | str` (`transcript.py:431`), so `7` (int) and `"7"` (str) never collide; `chat_id_json` round-trips the original typed id.

   **Byte-identity requirement (S4).** Rehydrate is cache-safe only if the reconstructed message dicts serialize to the **same request bytes** as before the restart. The model path hands dicts straight to the SDK (`query_engine.py:1213`), so what must match is the *final request serialization*, not the DB payload. Persist each message with a canonical `json.dumps(..., ensure_ascii=False, separators=(",", ":"))` that **preserves key insertion order** (no `sort_keys`), and reconstruct dicts preserving order on load. Lock it with a **byte-level rehydrate test**: assemble a request from a live store, restart-rehydrate from `transcripts.db`, re-assemble, and assert the serialized request-prefix bytes are identical.

3. **Operations mirror every store mutation.**
   - **Append** (`append_user_message` / `append_assistant_message` / `append_tool_message`, `transcript.py:473-507`) = insert the next `seq`.
   - **Collapse** (`replace_history`, `transcript.py:448`, called on compaction persist at `query_engine.py:637`) = `BEGIN IMMEDIATE`; `DELETE` the chat's rows; `INSERT [summary, *survivors]` at `seq` `0..n`; `UPDATE last_seq`; `COMMIT` — one atomic re-warm mirror.
   - **Rehydrate** = `SELECT payload_json … ORDER BY seq`.
   - **Sanitize writeback.** `prepare_history` rewrites history in place (`transcript.py:522-524`, `history[:] = self.sanitizer(...)`) and is usually a no-op (idempotent). Detect the no-op (compare before/after) and **skip the write** in the common case; when sanitize actually repairs an interrupted tool bundle it may **insert** placeholder messages mid-list and shift `seq`s, so mirror that as a **bounded replace** (delete + reinsert the affected suffix in one transaction, like collapse), never an append. Do not rely on re-deriving sanitize on load — the repaired bytes are the P-state and must be what the DB holds.

4. **Rehydrate cadence = one-time, on in-memory miss only.** Rehydrate when `get_or_create` finds no in-memory entry — i.e. cold-start after restart, or after LRU eviction of a still-persisted chat — then continue **append-only** in memory. **Never reload per-turn:** the in-memory `TranscriptStore` remains the sole source of truth for building requests; the DB is a durability + cold-start backstop. A per-turn reload would reserialize history and break ADR 0024's Stable prefix invariant. **PROPOSED — confirm:** trigger rehydrate lazily inside `get_or_create` on miss (recommended), rather than eagerly warming all chats at boot — lazy keeps startup O(1) and only pays I/O for revisited conversations.

5. **Durability posture: synchronous write-through via stdlib `sqlite3`.** Because `TranscriptStore`'s mutation API is **synchronous** (`transcript.py:473-507`), the mirror uses the standard-library **`sqlite3`** driver (sync, zero new dependency), **not** async `aiosqlite` — `aiosqlite` would require `await conn.commit()` inside a sync method, a contradiction (and would force the whole append path async). Open one write connection in **WAL** mode with a `busy_timeout`, guard writes with a per-store lock (SQLite serialises writers anyway), allocate `seq` atomically from `last_seq` inside the write transaction, and **commit each mutation before the store method returns**. This gives **true write-through durability**: any *committed* mutation survives a hard crash (`kill -9`); the only loss window is a mutation inside its own commit. **Honest latency:** the commit **is on the turn path** — appends fire at user-turn start (`query_engine.py:1337`), after each tool (`query_engine.py:877`), and before the assistant reply returns (`query_engine.py:1476`) — not "off the hot path". A small WAL commit is *expected* to be sub-millisecond and small next to the LLM/tool latency already on that turn, but **this must be benchmarked, not asserted**. If it proves material, a bounded async writer may be reintroduced **only with** a per-chat flush-before-evict / flush-before-rehydrate barrier and an explicit "loss = un-flushed backlog" statement — never as the silent default. **This supersedes the earlier (2026-07-03) ratification of an async background writer with block-on-overflow backpressure** (review finding **B3**: that design could lose a whole turn's multi-mutation backlog on `kill -9`).

6. **Split memory eviction from conversation deletion (this changes today's behaviour).** Today, LRU eviction **deletes** a chat's tool blobs: `_evict_lru_conversations` (`session.py:435-445`) calls `tool_result_store.clear(chat_id)` → `shutil.rmtree(...)` (`tool_result.py:77-79`), locked by `tests/bot/test_session.py:218`. That is fatal for rehydrate — a chat evicted from memory and later revisited would rehydrate transcript rows whose `full_result_path` blobs were deleted. This ADR therefore **redefines eviction as memory-only**:
   - **LRU eviction** (`transcript.py:462-471`, `session.py:435-445`) clears the in-memory working set **only** — it **must no longer delete tool blobs**, and it keeps both the `transcripts.db` rows and the `ToolResultStore` blob files, so a revisit rehydrates a consistent transcript with intact blobs. The eviction call to `tool_result_store.clear` is **removed**; `tests/bot/test_session.py:218` is updated to assert blobs survive eviction.
   - **`clear(chat_id)`** (user `/clear` / `/forget`, `transcript.py:444-446`) is the **only** path that deletes durable state: it removes the `transcripts.db` rows **and** the `ToolResultStore` blobs together (no orphans).
   - **New retention obligation.** Removing the eviction-time delete means the on-disk blobs *and* the in-memory `ToolResultStore.records_by_chat` map (`tool_result.py:59`) are no longer bounded by eviction. Two rules (full design in Open — a pre-launch requirement): (a) the in-memory `records_by_chat` entry **may** still be LRU-evicted to cap memory, but the **file stays** — the transcript message's `full_result_path` is the durable reference. **Caveat:** loading a body today needs a `ToolResultRecord` (`load_full_content`, `tool_result.py:84-87`) and compaction reads `records_by_chat` (`compaction.py:616-620`), so evicting the record cache requires **either** a path-based loader **or** reconstructing the record from the rehydrated ref — "records_by_chat is just a cache" is a target, not current behaviour. (b) blob deletion must be **reference-aware and lifecycle-safe**: a blob is written *before* its transcript ref is appended (`tool_result.py:156-164` → `query_engine.py:868-877`), so a naive "scan refs, delete unreferenced" GC could delete a just-written file — use an age grace / staging marker / shared lifecycle lock, and scan **all** live payload text (including `system` collapse summaries), which is safe only if ADR 0039's fidelity gate preserves every `full_result_path` through collapse.
   - Because every mutation is committed synchronously (Decision 5), an evicted chat has no un-flushed writes, so rehydrate always sees a consistent DB state.
   - Tool blobs stay file-based in `ToolResultStore`: structured messages live in `transcripts.db`, large blob bodies stay in files referenced by `full_result_path`. The DB does not absorb blob bytes.

### Invariants preserved

This ADR **keeps ADR 0024's prefix-cache invariants in full.** Because rehydrate restores the exact P-state bytes and happens exactly once on miss, the first `prepare_history` after a restart produces **byte-identical** output to the pre-restart turn — so the **Frozen tool list**, the **Session prefix snapshot**, and the append-only history all reproduce, and the **Stable prefix invariant** holds *across restarts*, not just within one process. No per-turn reload is introduced, so no new prefix reserialization exists. The two-tier `placement` boundary, append-only-between-collapses history, context collapse as the sole overflow handler, and the target-ratio < trigger-ratio byte-stability of collapse are all untouched: this ADR adds a durability mirror beneath the existing model path, changing none of it. (ADR 0024's separate char-budget open item — `max_prompt_chars` / `DEFAULT_MAX_PROMPT_CHARS` — is out of scope here and is superseded by ADR 0039's token-native budget.)

## Consequences

### Positive

- The last in-memory-only store gains restart/redeploy durability, closing F14 and unblocking the multi-user, restart-resilient backend goal. A graceful redeploy no longer amnesiac-resets live conversations.
- Rehydrate is byte-exact and one-shot, so ADR 0024's cache design survives restarts intact — a rehydrated session keeps hitting the provider prefix cache from turn 2 on, exactly as before the restart.
- The mirror reuses existing seams (`append_*`, `replace_history`, `prepare_history`, `clear`) and the standard-library `sqlite3` (no new dependency); no new model-path code, no change to request assembly.
- Opaque JSON payloads make the store format-agnostic: message-shape changes (new keys, tool-reference formats) need no schema migration.

### Negative

- Every store mutation now performs a synchronous WAL commit **on the turn path** (appends fire during turn processing, not off it). A WAL commit of a small row is *expected* to be small next to the LLM/tool latency on the same turn, but this must be **benchmarked, not assumed** (Decision 5); the cost is a bounded per-append I/O.
- Durability is **true write-through**: any committed mutation survives a hard crash; only a mutation inside its own sub-ms commit can be lost. (Revised from the earlier async-writer design per review finding B3.)
- Eviction behaviour **changes**: LRU eviction no longer deletes tool blobs (it does today) — only `/clear` deletes durable state; the test that asserted eviction purges blobs is updated (finding B2).
- Two stores must stay lifecycle-aligned (`clear` fans out to both); a future store added to the chat lifecycle must join the same fan-out or risk orphans.
- Disk grows with retained conversations (see Open — GC is a pre-launch item).

### Open

- **Cross-process sharing** — a shared `transcripts.db` across multiple backend workers (locking, WAL contention, who owns rehydrate). Deferred: single backend process for now.
- **Retention / GC (blocking for multi-user launch).** Removing eviction-time deletion (Decision 6) removes the *only* current bound on tool-blob disk use **and** on the in-memory `records_by_chat` map (`tool_result.py:59`). Durability is the near-term goal, but before a real multi-user deployment a retention design must land: an age/size cap on chats; a bound on `records_by_chat` (LRU-evict the metadata cache while keeping files — which requires a **path-based loader or record reconstruction**, since `load_full_content` needs a record today, `tool_result.py:84-87`); **reference-aware, lifecycle-safe blob GC** (scan **all** live payload refs incl. `system` collapse summaries before deleting; never delete an in-flight blob written before its ref is appended — use an age grace / staging marker / lifecycle lock); and periodic DB vacuum. This is a **pre-launch requirement, not an indefinite defer** — an unbounded `transcripts.db` plus unbounded blobs grow with every conversation ever held.
- **App-side "restored N turns" indicator** — surfacing that a conversation was rehydrated. Deferred: the App is untouched in this step — persistence is backend-transparent by design.
- **Lazy-on-miss rehydrate (Decision 4)** — RESOLVED as shipped (rehydrate fires only inside `get_or_create` on an in-memory miss; never per-turn). Decision 5's durability posture was revised to **synchronous write-through** per review finding B3, superseding the earlier async-writer / block-on-overflow ratification.
- **Blob-GC fidelity precondition (Decision 6) — partially hardened (2026-07-06).** D6 states reference-aware blob GC is safe only if collapse preserves every `full_result_path`. The collapse summary now recognizes the **micro-compaction** marker (`omicsclaw/runtime/storage/transcript.py`) so micro-compacted paths survive collapse, and ADR 0039's B1 gate preserves them through an LLM summary. **Still open:** the template caps tool refs at `max_compacted_refs` (default 3), so a collapse over >3 blob-backed results can still drop refs 4..n — the reference-aware GC (when built) must account for this or the cap must be lifted for GC-safety.

## Relationship to prior ADRs

- **ADR 0024** (prompt-prefix caching) — **prefix-cache invariants preserved in full.** This ADR adds durability *beneath* the model path and asserts byte-identical rehydrate, so the Stable prefix invariant, Frozen tool list, two-tier `placement`, append-only history, and collapse-as-sole-overflow all continue to hold — now across restarts. It introduces **no** per-turn reload. ADR 0024's char-budget open item (`max_prompt_chars` / `DEFAULT_MAX_PROMPT_CHARS`) is *not* preserved here — it is superseded by ADR 0039's token-native budget, out of scope for this durability mirror.
- **ADR 0039** (token-native context budget) — the **forcing function** for Decision 1 is its Decision 5, the LLM condensed collapse summary (**default-ON target**, gated on its fidelity-gate prerequisite): because the collapse summary bytes are LLM-generated and non-deterministic whenever enabled, the store must persist the derived P-state, never re-derive it on load.
- **Context-assembly review F14** — implements the deferred F14 write-through + cold-start rehydrate, explicitly **without** the rejected per-turn DB fetch, which the review already flagged as conflicting with prefix caching.
