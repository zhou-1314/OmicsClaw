11: # OmicsClaw Skill Guide â€” Spatial Preprocess
12: 
13: **Status**: implementation-aligned guide derived from the current OmicsClaw
14: `spatial-preprocess` skill. This is **not** one of the 28 already validated
15: end-to-end workflows. It is a living guide for preprocessing parameter
16: reasoning, loader expectations, and wrapper-specific caveats.
17: 
18: ## Purpose
19: 
20: Use this guide when you need to decide:
21: - how to explain the current `scanpy_standard` preprocessing workflow
22: - which QC and graph parameters matter first for a spatial dataset
23: - how to distinguish wrapper-level loader controls from actual preprocessing controls
24: 
25: ## Step 1: Inspect The Data First
26: 
27: If the dataset has not been inspected yet in this conversation, call
28: `inspect_data` first.
29: 
30: Key properties to check:
31: - **Input type**:
32:   - `raw_counts.h5ad` emitted by `spatial-raw-processing`
33:   - Visium directory or feature matrix
34:   - Xenium `.zarr` / `.h5ad`
35:   - converted `.h5ad` for MERFISH / Slide-seq / seqFISH
36:   - if the user still has FASTQ pairs plus IDs / STAR reference files, route to `spatial-raw-processing` instead
37: - **Matrix representation**:
38:   - current wrapper expects raw counts in `X` on input
39:   - if the input already looks log-normalized or scaled, say that explicitly before preprocessing
40: - **Spatial coordinates**:
41:   - coordinates are strongly preferred
42:   - preprocessing can still run without them, but downstream spatial plots and analyses will be limited
43: - **Species**:
44:   - current wrapper only exposes `human` and `mouse`
45:   - this matters for mitochondrial prefix detection (`MT-` vs `mt-`)
46: - **Dataset size and sparsity**:
47:   - very sparse datasets interact strongly with `min_genes`, `max_mt_pct`, and `n_top_hvg`
48: - **Biological context**:
49:   - if the tissue is obvious, a tissue preset may be a reasonable first-pass shortcut
50: 
51: Important implementation notes in current OmicsClaw:
52: - The current skill exposes one implemented workflow: `scanpy_standard`.
53: - `data_type`, `species`, and `tissue` are wrapper-level controls, not alternative preprocessing algorithms.
54: - Tissue presets only fill thresholds when the user has not already supplied explicit values.
55: - Reports now record the **effective** thresholds after preset application.
56: - Raw counts are preserved in both `adata.layers["counts"]` and `adata.raw`.
57: - PCA is computed on the HVG subset, and the neighbor graph uses a clipped PC count that may be lower than the requested count if the data shape requires it.
58: 
59: ## Step 2: Explain The Current Workflow Correctly
60: 
61: Current OmicsClaw `spatial-preprocess` runs:
62: 1. QC metric calculation with mitochondrial annotation.
63: 2. Spot / gene filtering.
64: 3. Raw-count preservation.
65: 4. Library-size normalization and `log1p`.
66: 5. `seurat_v3` highly variable gene selection on the counts layer.
67: 6. Scaling, PCA, neighbors, UMAP, and Leiden clustering.
68: 
69: Do **not** describe this as if the user were choosing among multiple
70: implemented preprocessing backends. Right now, the choice is mainly about
71: thresholds, graph construction, and loader hints.
72: 
73: ## Step 3: Pick Parameters Deliberately
74: 
75: Use this quick guide when the user has not specified exact thresholds:
76: 
77: | Parameter group | Best first use | Strong starting point | Main caveat |
78: |-----------------|----------------|-----------------------|-------------|
79: | **Tissue preset** | Fast first pass when tissue identity is obvious | `tissue=brain`, `tissue=tumor`, etc. | Presets are wrapper-level shortcuts, not universal truth |
80: | **QC thresholds** | Core quality filtering | start with preset or `min_genes=200`, `max_mt_pct=10~20` depending on tissue | Over-filtering can remove biologically real regions |
81: | **HVG budget** | Feature selection for PCA / graph | `n_top_hvg=2000` | Too low can oversimplify; too high can add noise |
82: | **Graph size** | neighborhood structure | `n_pcs=30`, `n_neighbors=15` | Optimal values depend on dataset size and heterogeneity |
83: | **Leiden resolution** | cluster granularity | `leiden_resolution=0.5` | Higher resolution is not automatically better |
84: | **Resolution sweep** | exploratory clustering | `resolutions=0.4,0.6,0.8,1.0` | Useful for exploration, not always needed in the first pass |
85: 
86: ## Step 4: Always Show An Effective Parameter Summary Before Running
87: 
88: Before execution, tell the user what will be run in a short, concrete block:
89: 
90: ```text
91: About to run spatial preprocessing
92:   Workflow: scanpy_standard
93:   Loader: data_type=visium
94:   Effective QC: min_genes=200, min_cells=0, max_mt_pct=10, max_genes=6000
95:   Graph: n_top_hvg=2000, n_pcs=30, n_neighbors=15, leiden_resolution=0.5
96:   Note: brain preset was used to fill QC defaults, but these effective thresholds are what will actually be applied.
97: ```
98: 
99: This is important because saying only "I will use the brain preset" is not
100: precise enough for reproducibility.
101: 
102: ## Step 5: Parameter-Specific Tuning Rules
103: 
104: ### Tissue preset
105: 
106: Tune in this order:
107: 1. `tissue`
108: 2. explicit threshold overrides
109: 
110: Guidance:
111: - Use a tissue preset only when the tissue identity is reasonably clear.
112: - If the user already knows the right QC regime, explicit numeric thresholds are better than relying entirely on a preset.
113: - If a preset is used, explain the resolved values before execution.
114: 
115: Important warning:
116: - Tissue presets are OmicsClaw wrapper knowledge, not upstream Scanpy APIs.
117: 
118: ### QC thresholds
119: 
120: Tune in this order:
121: 1. `min_genes`
122: 2. `max_mt_pct`
123: 3. `max_genes`
124: 4. `min_cells`
125: 
126: Guidance:
127: - Raise `min_genes` when extremely sparse spots are dominating the dataset.
128: - Lower `min_genes` when sparse biology is expected and the user accepts more noise.
129: - Lower `max_mt_pct` when mitochondrial-rich low-quality spots are obvious.
130: - Raise `max_mt_pct` cautiously in tissues like heart or muscle where higher mitochondrial load may be biological.
131: - Use `max_genes` as an upper-bound guard when obvious doublet-like or overly complex spots are a concern.
132: - Use `min_cells` when you want to remove genes detected in very few spots before downstream modeling.
133: 
134: Important warning:
135: - Aggressive QC may remove real tissue niches, especially in sparse or damaged specimens.
136: 
137: ### HVG budget
138: 
139: Tune in this order:
140: 1. `n_top_hvg`
141: 
142: Guidance:
143: - Start with `n_top_hvg=2000`.
144: - Increase it when the dataset is large and biologically diverse.
145: - Reduce it when the dataset is small or noisy and the user wants a tighter feature set.
146: 
147: Important warning:
148: - In current OmicsClaw, HVG selection uses `flavor="seurat_v3"` on the counts layer, not on log-normalized `adata.X`.
149: 
150: ### PCA and graph construction
151: 
152: Tune in this order:
153: 1. `n_pcs`
154: 2. `n_neighbors`
155: 
156: Guidance:
157: - Start with `n_pcs=30` and `n_neighbors=15`.
158: - Larger `n_pcs` can help when the dataset is complex, but the wrapper may clip the effective number based on matrix shape.
159: - Lower `n_neighbors` when you want more local structure.
160: - Raise `n_neighbors` when you want smoother, broader neighborhoods.
161: - After the run, compare requested PCs to computed / used / suggested PCs instead of assuming they were identical.
162: 
163: Important warning:
164: - Requested PCs and effective PCs are not always the same in the current wrapper.
165: 
166: ### Leiden resolution
167: 
168: Tune in this order:
169: 1. `leiden_resolution`
170: 2. `resolutions`
171: 
172: Guidance:
173: - Start with `leiden_resolution=0.5`.
174: - Increase it when the current clustering is too coarse.
175: - Lower it when clusters are obviously over-split.
176: - Use `resolutions` when the user explicitly wants exploratory granularity comparisons.
177: 
178: Important warning:
179: - A larger number of clusters is not automatically a better preprocessing result.
180: 
181: ## Step 6: What To Say After The Run
182: 
183: - If very few spots remain after QC: mention that thresholds were likely too strict and point to `min_genes`, `max_mt_pct`, or `max_genes`.
184: - If no spatial coordinates are present: explain that preprocessing succeeded, but spatial plots and downstream spatial analyses may be limited.
185: - If clustering is too coarse: suggest revisiting `leiden_resolution` or `n_neighbors`.
186: - If clustering is too fragmented: suggest lowering `leiden_resolution` or increasing `n_neighbors`.
187: - If requested PCs were much larger than used PCs: explain that the wrapper clipped the effective dimensionality based on the available HVGs and cells.
188: - If the dataset appears already normalized on input: mention that the preprocessing assumptions may be violated and the user should confirm whether rerunning normalization is appropriate.
189: 
190: ## Step 7: Explain Outputs Correctly
191: 
192: When summarizing results:
193: - describe `processed.h5ad` as the downstream-ready AnnData
194: - describe `tables/qc_summary.csv` as the compact summary of retained size and PCA usage
195: - describe `tables/cluster_summary.csv` as the primary Leiden cluster sizes
196: - describe `tables/multi_resolution_summary.csv` as optional exploratory output when a resolution sweep was run
197: - describe `effective_params` in `result.json` and `report.md` as the actual thresholds and graph settings that governed the preprocessing run
198: 
199: Do **not** say "the preset was brain, so preprocessing is reproducible" without
200: also stating the resolved thresholds. In current OmicsClaw, the effective
201: values are what matter for interpretation and reruns.
