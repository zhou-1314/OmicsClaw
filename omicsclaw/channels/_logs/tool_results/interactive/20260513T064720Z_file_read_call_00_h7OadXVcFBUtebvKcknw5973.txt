14: 11: # OmicsClaw Skill Guide — Spatial Preprocess
15: 12: 
16: 13: **Status**: implementation-aligned guide derived from the current OmicsClaw
17: 14: `spatial-preprocess` skill. This is **not** one of the 28 already validated
18: 15: end-to-end workflows. It is a living guide for preprocessing parameter
19: 16: reasoning, loader expectations, and wrapper-specific caveats.
20: 17: 
21: 18: ## Purpose
22: 19: 
23: 20: Use this guide when you need to decide:
24: 21: - how to explain the current `scanpy_standard` preprocessing workflow
25: 22: - which QC and graph parameters matter first for a spatial dataset
26: 23: - how to distinguish wrapper-level loader controls from actual preprocessing controls
27: 24: 
28: 25: ## Step 1: Inspect The Data First
29: 26: 
30: 27: If the dataset has not been inspected yet in this conversation, call
31: 28: `inspect_data` first.
32: 29: 
33: 30: Key properties to check:
34: 31: - **Input type**:
35: 32:   - `raw_counts.h5ad` emitted by `spatial-raw-processing`
36: 33:   - Visium directory or feature matrix
37: 34:   - Xenium `.zarr` / `.h5ad`
38: 35:   - converted `.h5ad` for MERFISH / Slide-seq / seqFISH
39: 36:   - if the user still has FASTQ pairs plus IDs / STAR reference files, route to `spatial-raw-processing` instead
40: 37: - **Matrix representation**:
41: 38:   - current wrapper expects raw counts in `X` on input
42: 39:   - if the input already looks log-normalized or scaled, say that explicitly before preprocessing
43: 40: - **Spatial coordinates**:
44: 41:   - coordinates are strongly preferred
45: 42:   - preprocessing can still run without them, but downstream spatial plots and analyses will be limited
46: 43: - **Species**:
47: 44:   - current wrapper only exposes `human` and `mouse`
48: 45:   - this matters for mitochondrial prefix detection (`MT-` vs `mt-`)
49: 46: - **Dataset size and sparsity**:
50: 47:   - very sparse datasets interact strongly with `min_genes`, `max_mt_pct`, and `n_top_hvg`
51: 48: - **Biological context**:
52: 49:   - if the tissue is obvious, a tissue preset may be a reasonable first-pass shortcut
53: 50: 
54: 51: Important implementation notes in current OmicsClaw:
55: 52: - The current skill exposes one implemented workflow: `scanpy_standard`.
56: 53: - `data_type`, `species`, and `tissue` are wrapper-level controls, not alternative preprocessing algorithms.
57: 54: - Tissue presets only fill thresholds when the user has not already supplied explicit values.
58: 55: - Reports now record the **effective** thresholds after preset application.
59: 56: - Raw counts are preserved in both `adata.layers["counts"]` and `adata.raw`.
60: 57: - PCA is computed on the HVG subset, and the neighbor graph uses a clipped PC count that may be lower than the requested count if the data shape requires it.
61: 58: 
62: 59: ## Step 2: Explain The Current Workflow Correctly
63: 60: 
64: 61: Current OmicsClaw `spatial-preprocess` runs:
65: 62: 1. QC metric calculation with mitochondrial annotation.
66: 63: 2. Spot / gene filtering.
67: 64: 3. Raw-count preservation.
68: 65: 4. Library-size normalization and `log1p`.
69: 66: 5. `seurat_v3` highly variable gene selection on the counts layer.
70: 67: 6. Scaling, PCA, neighbors, UMAP, and Leiden clustering.
71: 68: 
72: 69: Do **not** describe this as if the user were choosing among multiple
73: 70: implemented preprocessing backends. Right now, the choice is mainly about
74: 71: thresholds, graph construction, and loader hints.
75: 72: 
76: 73: ## Step 3: Pick Parameters Deliberately
77: 74: 
78: 75: Use this quick guide when the user has not specified exact thresholds:
79: 76: 
80: 77: | Parameter group | Best first use | Strong starting point | Main caveat |
81: 78: |-----------------|----------------|-----------------------|-------------|
82: 79: | **Tissue preset** | Fast first pass when tissue identity is obvious | `tissue=brain`, `tissue=tumor`, etc. | Presets are wrapper-level shortcuts, not universal truth |
83: 80: | **QC thresholds** | Core quality filtering | start with preset or `min_genes=200`, `max_mt_pct=10~20` depending on tissue | Over-filtering can remove biologically real regions |
84: 81: | **HVG budget** | Feature selection for PCA / graph | `n_top_hvg=2000` | Too low can oversimplify; too high can add noise |
85: 82: | **Graph size** | neighborhood structure | `n_pcs=30`, `n_neighbors=15` | Optimal values depend on dataset size and heterogeneity |
86: 83: | **Leiden resolution** | cluster granularity | `leiden_resolution=0.5` | Higher resolution is not automatically better |
87: 84: | **Resolution sweep** | exploratory clustering | `resolutions=0.4,0.6,0.8,1.0` | Useful for exploration, not always needed in the first pass |
88: 85: 
89: 86: ## Step 4: Always Show An Effective Parameter Summary Before Running
90: 87: 
91: 88: Before execution, tell the user what will be run in a short, concrete block:
92: 89: 
93: 90: ```text
94: 91: About to run spatial preprocessing
95: 92:   Workflow: scanpy_standard
96: 93:   Loader: data_type=visium
97: 94:   Effective QC: min_genes=200, min_cells=0, max_mt_pct=10, max_genes=6000
98: 95:   Graph: n_top_hvg=2000, n_pcs=30, n_neighbors=15, leiden_resolution=0.5
99: 96:   Note: brain preset was used to fill QC defaults, but these effective thresholds are what will actually be applied.
100: 97: ```
101: 98: 
102: 99: This is important because saying only "I will use the brain preset" is not
103: 100: precise enough for reproducibility.
104: 101: 
105: 102: ## Step 5: Parameter-Specific Tuning Rules
106: 103: 
107: 104: ### Tissue preset
108: 105: 
109: 106: Tune in this order:
110: 107: 1. `tissue`
111: 108: 2. explicit threshold overrides
112: 109: 
113: 110: Guidance:
114: 111: - Use a tissue preset only when the tissue identity is reasonably clear.
115: 112: - If the user already knows the right QC regime, explicit numeric thresholds are better than relying entirely on a preset.
116: 113: - If a preset is used, explain the resolved values before execution.
117: 114: 
118: 115: Important warning:
119: 116: - Tissue presets are OmicsClaw wrapper knowledge, not upstream Scanpy APIs.
120: 117: 
121: 118: ### QC thresholds
122: 119: 
123: 120: Tune in this order:
124: 121: 1. `min_genes`
125: 122: 2. `max_mt_pct`
126: 123: 3. `max_genes`
127: 124: 4. `min_cells`
128: 125: 
129: 126: Guidance:
130: 127: - Raise `min_genes` when extremely sparse spots are dominating the dataset.
131: 128: - Lower `min_genes` when sparse biology is expected and the user accepts more noise.
132: 129: - Lower `max_mt_pct` when mitochondrial-rich low-quality spots are obvious.
133: 130: - Raise `max_mt_pct` cautiously in tissues like heart or muscle where higher mitochondrial load may be biological.
134: 131: - Use `max_genes` as an upper-bound guard when obvious doublet-like or overly complex spots are a concern.
135: 132: - Use `min_cells` when you want to remove genes detected in very few spots before downstream modeling.
136: 133: 
137: 134: Important warning:
138: 135: - Aggressive QC may remove real tissue niches, especially in sparse or damaged specimens.
139: 136: 
140: 137: ### HVG budget
141: 138: 
142: 139: Tune in this order:
143: 140: 1. `n_top_hvg`
144: 141: 
145: 142: Guidance:
146: 143: - Start with `n_top_hvg=2000`.
147: 144: - Increase it when the dataset is large and biologically diverse.
148: 145: - Reduce it when the dataset is small or noisy and the user wants a tighter feature set.
149: 146: 
150: 147: Important warning:
151: 148: - In current OmicsClaw, HVG selection uses `flavor="seurat_v3"` on the counts layer, not on log-normalized `adata.X`.
152: 149: 
153: 150: ### PCA and graph construction
154: 151: 
155: 152: Tune in this order:
156: 153: 1. `n_pcs`
157: 154: 2. `n_neighbors`
158: 155: 
159: 156: Guidance:
160: 157: - Start with `n_pcs=30` and `n_neighbors=15`.
161: 158: - Larger `n_pcs` can help when the dataset is complex, but the wrapper may clip the effective number based on matrix shape.
162: 159: - Lower `n_neighbors` when you want more local structure.
163: 160: - Raise `n_neighbors` when you want smoother, broader neighborhoods.
164: 161: - After the run, compare requested PCs to computed / used / suggested PCs instead of assuming they were identical.
165: 162: 
166: 163: Important warning:
167: 164: - Requested PCs and effective PCs are not always the same in the current wrapper.
168: 165: 
169: 166: ### Leiden resolution
170: 167: 
171: 168: Tune in this order:
172: 169: 1. `leiden_resolution`
173: 170: 2. `resolutions`
174: 171: 
175: 172: Guidance:
176: 173: - Start with `leiden_resolution=0.5`.
177: 174: - Increase it when the current clustering is too coarse.
178: 175: - Lower it when clusters are obviously over-split.
179: 176: - Use `resolutions` when the user explicitly wants exploratory granularity comparisons.
180: 177: 
181: 178: Important warning:
182: 179: - A larger number of clusters is not automatically a better preprocessing result.
183: 180: 
184: 181: ## Step 6: What To Say After The Run
185: 182: 
186: 183: - If very few spots remain after QC: mention that thresholds were likely too strict and point to `min_genes`, `max_mt_pct`, or `max_genes`.
187: 184: - If no spatial coordinates are present: explain that preprocessing succeeded, but spatial plots and downstream spatial analyses may be limited.
188: 185: - If clustering is too coarse: suggest revisiting `leiden_resolution` or `n_neighbors`.
189: 186: - If clustering is too fragmented: suggest lowering `leiden_resolution` or increasing `n_neighbors`.
190: 187: - If requested PCs were much larger than used PCs: explain that the wrapper clipped the effective dimensionality based on the available HVGs and cells.
191: 188: - If the dataset appears already normalized on input: mention that the preprocessing assumptions may be violated and the user should confirm whether rerunning normalization is appropriate.
192: 189: 
193: 190: ## Step 7: Explain Outputs Correctly
194: 191: 
195: 192: When summarizing results:
196: 193: - describe `processed.h5ad` as the downstream-ready AnnData
197: 194: - describe `tables/qc_summary.csv` as the compact summary of retained size and PCA usage
198: 195: - describe `tables/cluster_summary.csv` as the primary Leiden cluster sizes
199: 196: - describe `tables/multi_resolution_summary.csv` as optional exploratory output when a resolution sweep was run
200: 197: - describe `effective_params` in `result.json` and `report.md` as the actual thresholds and graph settings that governed the preprocessing run
201: 198: 
202: 199: Do **not** say "the preset was brain, so preprocessing is reproducible" without
203: 200: also stating the resolved thresholds. In current OmicsClaw, the effective
204: 201: values are what matter for interpretation and reruns.
