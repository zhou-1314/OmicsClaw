# Unified Skill Optimization Template

> **Status**: Reusable optimization blueprint for OmicsClaw skills
> **Last Updated**: 2026-03-30
> **Applies To**: `skills/spatial/*` first, then other domains under `skills/`

## 1. Purpose

This document is a **cross-skill optimization specification**, not a replacement
for [`templates/SKILL-TEMPLATE.md`](../templates/SKILL-TEMPLATE.md).

Use this document when you want to **upgrade an existing skill** so it becomes:

- easier for users to understand and trust
- more accurate in method selection and parameter explanation
- more reproducible in CLI, bot, and interactive chat modes
- easier to extend when new methods are added later
- better aligned with the current OmicsClaw architecture

Use `templates/SKILL-TEMPLATE.md` when creating a new skill from scratch.
Use this document when **standardizing and hardening a real skill** that already
has code, methods, outputs, and knowledge dependencies.

## 2. Optimization Goals

Every optimized skill should satisfy the following five goals:

1. **Implementation-aligned**: documentation must describe what the wrapper
   actually does today, not only what the upstream paper claims.
2. **Method-aware**: different methods inside one skill must have different
   parameter exposure, defaults, and caveats.
3. **Prompt-safe**: short guardrails and long tuning guides must be separated so
   prompt injection stays concise.
4. **Notebook-first reproducibility**: successful runs should produce a readable
   reproducibility bundle, including `analysis_notebook.ipynb`, not only
   `commands.sh`.
5. **Extension-friendly**: adding a new method later should require local,
   method-scoped edits rather than rewriting the whole skill contract.

## 3. Scope And Layering

An optimized skill is not just one `SKILL.md` file. It is a coordinated package
across seven layers.

### Layer 1: `SKILL.md` Metadata Contract

Defines:

- the public description of the skill
- trigger keywords and aliases
- allowed CLI flags
- method-specific `param_hints`
- required inputs and output contract
- wrapper-specific limitations and behavior notes

### Layer 2: Wrapper Script Contract

Usually `skills/<domain>/<skill>/<skill>.py`.

Defines:

- CLI parsing
- mapping from public flags to backend method parameters
- report generation
- output directory layout
- reproducibility bundle generation
- notebook generation for normal CLI/bot/chat runs

### Layer 3: Backend Implementation Contract

Usually `skills/<domain>/_lib/<module>.py`.

Defines:

- true method execution
- method-specific defaults
- stable result keys written back to AnnData / tables
- environment-specific dependency handling
- method-aware error messages

### Layer 4: Output Contract

Defines:

- which files are always written
- which AnnData fields are guaranteed after success
- which figures/tables exist regardless of invocation path
- what users should inspect first

### Layer 5: Knowledge Contract

Split into two document types:

- `knowledge_base/knowhows/KH-<skill>-guardrails.md`
- `knowledge_base/skill-guides/<domain>/<skill>.md`

These two files have different responsibilities and must not be merged.

### Layer 6: Routing And Registration Contract

Usually includes:

- `omicsclaw/core/registry.py`
- `omicsclaw/knowledge/knowhow.py`
- keyword routing tests

This ensures the optimized skill is discoverable and the correct guardrails are
loaded in the current framework.

### Layer 7: Test Contract

At minimum:

- skill-specific tests
- lazy metadata tests
- keyword routing tests
- knowhow loading tests

## 4. Division Of Responsibilities

This boundary is critical and should be kept stable across domains.

| Artifact | Main role | Should contain | Should not contain |
|---|---|---|---|
| `SKILL.md` | execution contract | methods, flags, defaults, I/O, wrapper behavior | long tuning essays, verbose troubleshooting |
| `KH-*-guardrails.md` | short injected rules | concise pre-run / warning / post-run guardrails | full parameter encyclopedia |
| `skill-guides/<domain>/<skill>.md` | longer method guide | method choice, tuning order, interpretation, wrapper caveats | inflated prompt-injection content |
| validated workflows in `knowledge_base/` | stable analytical workflows | end-to-end curated procedures | rapidly changing wrapper notes |

In short:

- **`SKILL.md` answers**: what this skill is and how it runs
- **guardrails answer**: what the model must not forget during execution
- **skill guide answers**: how to choose and tune methods correctly
- **validated workflows answer**: what a trusted end-to-end standard workflow looks like

## 5. Public Parameter Design Rules

This is the most important part of the template.

### 5.1 Common Parameters vs Method-Specific Parameters

Expose only a small set of truly common wrapper parameters without prefixes.

Typical common parameters:

- `method`
- `n_top_*`
- `fdr_threshold`
- `cluster_key`
- `layer`
- `species`

Expose method-specific parameters with **method prefixes**.

Examples:

- `morans_n_neighs`
- `spatialde_min_counts`
- `sparkx_num_cores`
- `cellrank_n_states`
- `harmony_theta`

Rule:

- if a parameter only affects one method, it should usually be prefixed
- if a parameter affects all methods in the same semantic way, it can remain common

### 5.2 Only Expose Core Parameters

Do not dump every upstream argument into `allowed_extra_flags`.

**Before deciding which parameters to expose, perform a web search to understand the method's full parameter space.**

Expose only parameters that are:

1. scientifically important
2. likely to be adjusted by users
3. stable enough to support long-term
4. actually implemented in the current wrapper

Do not expose parameters that are:

- internal plumbing
- rarely useful debug knobs
- upstream-only but not wired into OmicsClaw
- unstable experiments that would confuse the user

### 5.3 Parameter Provenance Must Be Real (Web Search Required)

When exposing Method-Specific Parameters or Core Parameters, **you MUST actively query the web** to verify their exact names, types, and default values. 

For each exposed parameter, confirm its exact API definition from one of these sources:

1. official upstream documentation (via live web search)
2. actual callable API signatures (via source code inspection or web search)
3. wrapper-level logic implemented in OmicsClaw

**CRITICAL RULE**: Do not hallucinate or guess parameter names based on general knowledge. You MUST perform a live web search for the target method's official documentation before finalizing `allowed_extra_flags`, `param_hints`, or backend code. Do not invent generic `param_hints` that look plausible but are not tied to a real method interface.

### 5.4 Wrapper-Level Knobs Must Be Labeled Honestly

Some parameters are not official upstream method parameters, but are still valid
OmicsClaw controls.

Examples:

- gene caps for runtime control
- wrapper-side bandwidth estimates
- sketch sizes for approximate computation
- plotting or export controls

These are allowed, but they must be described as:

- **wrapper-level controls**
- not guaranteed upstream public API knobs

### 5.5 `param_hints` Should Be Method-Specific

Recommended structure:

```yaml
param_hints:
  method_name:
    priority: "first_param -> second_param -> third_param"
    params: ["first_param", "second_param", "third_param"]
    defaults: {first_param: 1, second_param: 20}
    requires: ["obsm.spatial", "layers.counts"]
    tips:
      - "--first-param: Main tuning knob."
      - "--second-param: Runtime or granularity control."
```

Rules:

- one block per method
- parameter priority should reflect the real tuning order
- defaults should match actual wrapper defaults
- `requires` should reflect concrete data/environment requirements
- tips should be short and operational

## 6. Internal Code Design Rules

### 6.1 Single Source Of Truth For Defaults

Backend code should define method defaults centrally, for example:

```python
METHOD_PARAM_DEFAULTS = {
    "method_a": {...},
    "method_b": {...},
}
```

This should drive:

- runtime defaults
- `effective_params`
- documentation synchronization
- tests

### 6.2 `effective_params` Should Be Explicit

Every successful run should be able to report the **actual parameters used**.

Recommended pattern:

- merge wrapper defaults
- apply user overrides
- write `effective_params` into:
  - `result.json`
  - report text
  - notebook metadata when applicable

This reduces ambiguity in bot/CLI conversations and makes reruns easier.

### 6.3 Do Not Silently Fabricate Biological Structure

Never silently create fake biological metadata just to make a method run.

Examples of what should not happen silently:

- inventing cluster labels
- inventing root cells
- inventing batch labels
- pretending a missing raw-count layer exists

If a required input is missing:

- fail clearly, or
- use a documented fallback with an explicit warning

### 6.4 Do Not Silently Swap Methods

If the user selected method `A`, do not silently run method `B` and describe it
as method `A`.

Allowed behavior:

- fail with a clear dependency/input message
- suggest an alternative method

Not allowed:

- hidden substitution
- silently downgrading to another backend without telling the user

### 6.5 Persist Results Before Figure Generation

If `generate_figures(adata, output_dir)` depends on new embeddings, labels,
scores, or probabilities, those outputs must first be saved back into `adata`
or exported tables under stable keys.

Figure generation must not depend on transient local variables that disappear
after the analysis function returns.

### 6.6 Stable Result Keys Matter

For each method, define stable storage locations such as:

- `adata.obs[...]`
- `adata.obsm[...]`
- `adata.uns[...]`
- `tables/*.csv`

These keys must be:

- method-aware
- predictable
- documented in `SKILL.md` and tests

## 7. Output Contract

Optimized skills should converge on a readable, reproducible output layout.

Recommended baseline:

```text
output_dir/
├── README.md
├── report.md
├── result.json
├── processed.h5ad
├── figures/
├── tables/
├── figure_data/
└── reproducibility/
    ├── analysis_notebook.ipynb
    ├── commands.sh
    └── requirements.txt
```

Optional additions:

- `reproducibility/r_visualization.sh`
- `r_visualization/`
- method-specific extra tables

### 7.1 Minimum Semantics

- `README.md`: user-first navigation file; should tell the user where to start
- `report.md`: narrative explanation of findings and caveats
- `result.json`: machine-readable run summary
- `processed.h5ad`: output object with new annotations when applicable
- `figures/`: canonical quick-look layer
- `figure_data/`: downstream plotting contract
- `reproducibility/analysis_notebook.ipynb`: readable, stepwise rerun notebook
- `reproducibility/commands.sh`: compact shell replay entrypoint

### 7.2 Notebook Requirement

This is now an important user-experience rule:

- notebook generation should not be limited to multi-agent `/research` mode
- normal CLI execution and conversational execution should also produce a
  readable notebook when the wrapper succeeds

This keeps bot, CLI, and code-agent outputs aligned and makes analyses easier to
review and reproduce.

## 8. Knowledge Layer Template

Each optimized skill should add two knowledge files.

### 8.1 Guardrail File

Path pattern:

```text
knowledge_base/knowhows/KH-<skill>-guardrails.md
```

Purpose:

- short, high-value rules
- safe for forced injection into model context
- covers method choice, data inspection, and major interpretation warnings

Recommended content:

- inspect the right data first
- choose the method intentionally
- explain the method and key parameters before running
- use method-correct interpretation language
- point to the longer skill guide for deeper tuning

Do not put a full tuning encyclopedia here.

### 8.2 Skill Guide File

Path pattern:

```text
knowledge_base/skill-guides/<domain>/<skill>.md
```

Purpose:

- longer implementation-aware guide
- method choice logic
- tuning order per method
- wrapper-specific caveats
- result interpretation guidance
- large-dataset rules
- troubleshooting patterns

Recommended structure:

1. purpose
2. inspect the data first
3. pick the method deliberately
4. show a parameter summary before running
5. method-specific tuning rules
6. post-run interpretation rules
7. output / visualization contract

## 9. Registration And Routing Checklist

After optimizing a skill, update the framework wiring.

Typical checklist:

1. update `omicsclaw/core/registry.py`
2. update `omicsclaw/knowledge/knowhow.py`
3. ensure aliases and trigger keywords remain stable
4. ensure the new guardrail document is discoverable
5. ensure tests cover lazy metadata and keyword routing

Current framework note:

- the active knowhow loading path still uses explicit mapping logic
- therefore each newly optimized skill should be wired deliberately instead of
  assuming automatic full metadata discovery

## 10. Standard Optimization Workflow

Use this sequence when upgrading any existing skill.

### Step 1: Inventory The Skill

Read:

- `skills/<domain>/<skill>/SKILL.md`
- wrapper script
- relevant `_lib` backend modules
- existing tests

Identify:

- how many methods actually exist
- which outputs are already stable
- which method assumptions are undocumented
- which parameters are currently exposed but not justified

### Step 2: Verify Real Method Interfaces

For each method:

- inspect upstream docs or code signatures
- identify the real core parameters
- separate upstream parameters from wrapper-side controls
- record real data requirements and dependency requirements

### Step 3: Redesign The Public Contract

Rewrite `SKILL.md` so it includes:

- cleaner description
- clear workflow
- method-specific `allowed_extra_flags`
- method-specific `param_hints`
- true input matrix / metadata requirements
- output and reproducibility contract

### Step 4: Align Wrapper And Backend Code

Refactor code so that:

- defaults are centralized
- user overrides are merged cleanly
- results are written back under stable keys
- `generate_figures` reads persisted outputs
- failures become method-aware and interpretable

### Step 5: Add Knowledge Documents

Create:

- `KH-<skill>-guardrails.md`
- `skill-guides/<domain>/<skill>.md`

Ensure:

- guardrails stay short
- skill guide carries the longer method logic
- neither duplicates validated workflow documents unnecessarily

### Step 6: Wire Registry And Knowhow

Update:

- registry aliases and metadata
- knowhow injection mapping
- keyword routing expectations

### Step 7: Add Or Update Tests

Minimum test surface:

- CLI / parser behavior
- method dispatch behavior
- default propagation
- stable output keys
- lazy metadata loading
- keyword routing
- knowhow injection

### Step 8: Validate End To End

At minimum:

- `py_compile` for changed Python files
- targeted `pytest` for the skill and framework wiring tests
- one real smoke run if dependencies and demo data are available

## 11. Method Addition Recipe

When adding a new method to an already optimized skill, update all of the
following in one pass:

1. `allowed_extra_flags`
2. `param_hints`
3. parser / CLI arguments
4. `METHOD_PARAM_DEFAULTS`
5. backend dispatch logic
6. stable AnnData/table output keys
7. report text and `result.json`
8. guardrail wording if method choice changes materially
9. skill guide method section
10. tests

If one of these is skipped, the skill will drift out of sync.

## 12. Anti-Patterns To Avoid

- exposing the same vague parameter set for every method
- copying upstream paper text without checking current wrapper behavior
- putting long tuning manuals into guardrail docs
- describing wrapper-only knobs as official upstream parameters
- generating figures from transient runtime variables instead of persisted data
- silently falling back across methods without telling the user
- adding many flags without updating tests and documentation
- writing outputs that differ between `oc run`, bot execution, and interactive chat

## 13. Definition Of Done

An optimized skill is considered complete when all of the following are true:

- `SKILL.md` is implementation-aligned
- parameters are method-specific and justified
- wrapper code and backend code share a single default model
- outputs are stable and readable
- notebook reproducibility exists outside `/research` mode
- guardrail and skill-guide documents both exist and have clear boundaries
- registry and knowhow wiring are updated
- targeted tests pass

## 14. Recommended Reuse Pattern Across Domains

This template is intentionally domain-agnostic.

It can be reused for:

- spatial analysis skills
- single-cell analysis skills
- genomics skills
- proteomics skills
- metabolomics skills
- bulk RNA skills

What changes across domains:

- input object type
- method families
- required metadata
- downstream figures and tables

What should stay stable across domains:

- layered design
- method-specific parameter exposure
- short guardrails + longer skill guides
- notebook-first reproducibility
- stable output contracts
- registry / routing / tests as part of the optimization, not afterthoughts

## 15. Recommended Working Rule

When optimizing a skill, do not ask:

> "How can I make this file look nicer?"

Ask:

> "What is the smallest complete contract that makes this skill accurate,
> explainable, reproducible, and easy to extend?"

That question is the core of the OmicsClaw optimization template.
